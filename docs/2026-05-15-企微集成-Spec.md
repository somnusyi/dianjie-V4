## 一句话定位

**滇界是「业务系统」(数据/操作/决策闭环),企微是「触达通道」(SSO/通讯录/消息推送)。所有业务在滇界内部完成,企微不当 IM、不当文件库、不当审批引擎。**

---

## 一、分层架构

```
┌─────────────────────────────────────────────────────────┐
│  L4 · 入口聚合 (企微工作台 / 滇界 App)                    │
│      员工每天打开企微 → 工作台「滇界云管」 → SSO → hero   │
├─────────────────────────────────────────────────────────┤
│  L3 · 业务事件源 (滇界内部)                                │
│      订单/报损/库存/财务/异常 → 写 DB → 触发事件总线        │
├─────────────────────────────────────────────────────────┤
│  L2 · 触达层 (Notification 通道抽象)                      │
│      notify(event, recipients, payload)                  │
│      ├─ 企微 (内部员工 + 集团内供应商)                      │
│      ├─ SMS (外部供应商 / 紧急)                            │
│      └─ 滇界 app push (外部供应商 / 应用内 fallback)       │
├─────────────────────────────────────────────────────────┤
│  L1 · 身份层 (Identity)                                   │
│      企微 SSO / 通讯录单向同步 / 离职自动停用              │
│      企微 UserID ↔ 滇界 User 一对一                       │
└─────────────────────────────────────────────────────────┘
```

**设计原则**:
- **数据双源时,滇界为权威**(企微通讯录是输入,不覆盖滇界角色/权限)
- **通道可换**(明天换钉钉,L2 抽象不变,业务零修改)
- **触达有降级**(企微挂了 → SMS → app push,不丢)

## 二、用户身份分层(基于 ~401 人企微规模)

| 滇界角色 | 人数估 | 能力 | 默认企微推送 |
|---|---|---|---|
| **ADMIN** 老板 | 1-3 | 全局 | 大额支出、新店、人员重大变动、异常 |
| **FINANCE** 财务 | 2-5 | 资金/对账/发票 | 待付款、月度对账、异常流水 |
| **CHEF_DIRECTOR** 总厨 | 1-2 | 跨店食材标准/报损仲裁 | 报损争议、新品上架审批、食材成本异常 |
| **MANAGER** 店长 | N 店 × 1 | 本店一切 | 本店营收异常、本店待办 |
| **KITCHEN_LEAD** 厨师长 | N × 1 | 本店食材 | 订单送达、待验收、库存预警 |
| **SUPERVISOR** 主管 **(新)** | 5-10 | 本店有限事务 | 排班变更、本店异常通报 |
| **STAFF** 一线员工 **(新,留口)** | 0 (暂不开) | 仅企微触达,不进滇界 | — |
| **SUPPLIER** 供应商员工 | 20-50 | 自己订单/库存/账单 | 待接单、报损待回、账期通知 |

**总滇界用户库**: ~50-80 人(管理层 + 集团内供应商员工),其余 ~300 服务员**不开滇界账号**,纯企微触达个人事项。

### 权限维度

```
权限 = role                  // 基础角色
     + storeIds[]            // 多店支持
     + supplierId?           // 供应商员工绑定
     + featureFlags?         // 模块开关 (后期, 暂不实现)
     + status                // ACTIVE / INACTIVE / PENDING_BIND
```

## 三、4 类信息触达 + 防骚扰

| 类型 | 收件人 | 通道 | 频率上限 |
|---|---|---|---|
| **[A] 个人事项** (排班/工资/审批) | 仅本人 | 企微应用消息 | 日均 ≤ 2 |
| **[B] 角色事项** (待接单/审批/仲裁) | 角色组 1-5 人 | 企微卡片消息 | 实时+ 5min 防抖 |
| **[C] 店内事项** (营收异常/食材告警) | 店相关角色 3-10 人 | 企微应用消息 | 每店日均 ≤ 5 |
| **[D] 全员公告** (放假/培训) | 全员群发 | 企微通知 + 工作台 banner | 周 ≤ 2 |

### 防骚扰机制(M2 必带)

1. **用户级偏好**: 每事件类型可关 (`NotificationPref` 表)
2. **频控**: 同人同事件 5 分钟内不重复 (`NotificationLog.eventKey` 去重)
3. **静默时段**: 22:00 - 07:00 默认不推 (紧急除外)
4. **升级机制**: 首推该角色;1h 无人处理 → 上级角色

## 四、店长离职/调岗/换店设计

### 核心:**「当前归属」 vs 「历史归属」**

```
User.storeIds = [X, Y]    ← 当前管辖 (可变, 反映此刻角色)
Order.storeId  = X        ← 历史归属 (不变, 永远归 X 店)
*.createdById             ← 历史发起人 (不变)
```

**绝不动历史归属字段**。所有交接通过改 `User.storeIds` + `StoreHandover` 表记录实现。

### 4 种场景

| 场景 | 操作 | 待办处理 |
|---|---|---|
| **调岗** A: X→Y | A.storeIds [X]→[Y] | A 在 X 的未完事项交给接管者 |
| **离职** A 离职 | A.status=INACTIVE,storeIds=[],撤 token | 同上;必须先指定继任者再 INACTIVE |
| **接管** B 接 X | B.storeIds += [X] | B 自动继承"未处理"待办池 |
| **临时代管** A 临时管 Y | A.storeIds [X]+[Y] | 人工撤回,不做自动到期 |

### 数据模型

```prisma
model User {
  storeIds      String[]    @default([])    // 多店
  // 旧 storeId 通过 storeIds[0] 兼容
}

model StoreHandover {
  id            String   @id @default(cuid())
  tenantId      String
  storeId       String
  fromUserId    String
  toUserId      String
  reason        String?  // '调岗' / '离职' / '临时代管'
  handoverItems Json?    // [{type, id, status}, ...]
  createdAt     DateTime @default(now())
  @@index([tenantId, storeId, createdAt])
}
```

### 运维 UI

- 老板「门店管理」页:每店当前店长 + 历史交接链 + **无主门店预警**
- 离职流程:输入「继任者」→ 系统提示交接事项数 → 确认后才 INACTIVE

## 五、决策汇总表(讨论后锁定)

| 议题 | 决定 |
|---|---|
| 供应商通知通道 | 集团内子公司供应商走企微;外部供应商走滇界 app + SMS |
| OA(报销/请假) | 走企微原生,M5/M7 同步状态回滇界 |
| 部门→角色映射 | 按部门关键字猜 + 手动可覆盖 |
| 企微员工但滇界没账号 | 「待开通员工」列表,老板手动开通 |
| 离职处理 | INACTIVE + 撤销 token + 不动历史 |
| 数据合规 | phone/email 明文存,头像不存 URL 现取 |
| 服务员(~300 人) | 不开滇界账号,纯企微触达 |
| 排班/打卡/工资 | 不做,以后用企微生态(道一云/i 人事) |
| 多店店长 | `storeIds[]`,通过 StoreHandover 表记录交接 |
| 角色新增 | SUPERVISOR(主管)+ STAFF(留口) |

## 八、关键事件触达规则(M2 接通的 6 个 P0)

| EventType | 默认收件人 | 通道 | 升级 |
|---|---|---|---|
| `PO_SUBMITTED` 订单提交 | 该订单 supplier 全员 | 企微卡片 + SMS 备份 | 1h 无接单 → 重发 |
| `PO_DELIVERING` 发货 | 厨师长 + 店长 | 企微应用消息 | — |
| `PO_PENDING_CONFIRM` 到店 | 厨师长 | 企微卡片(高亮) | 4h 未验收 → 店长 |
| `LOSS_PENDING` 报损待处理 | 该订单 supplier | 企微卡片 + SMS | 24h 未响应 → 总厨 + 自动同意 |
| `LOSS_REJECTED` 供应商拒绝 | 总厨 | 企微卡片 | 24h 未仲裁 → 老板 |
| `PAYMENT_LARGE` 大额付款待审 | 财务 + 老板 | 企微卡片 | 4h 未审 → 重发 |

阈值参数化(全局可调):大额 ≥ ¥10,000,自动同意 24h,升级时长 1h/4h。

## 九、风险 & 待观察

| 风险 | 应对 |
|---|---|
| 推送频率高骚扰员工 | 频控 + 静默时段 + 用户偏好 |
| 企微 API 调用次数限制 | 当前用量小,1000 人企业 30 万次/日远超需求 |
| 企微账号被封 / corpsecret 泄漏 | 加密存储 + 立刻可旋转 secret |
| 业务事件丢失推送 | NotificationLog 全量审计,失败可重发 |
| 老板看到「全员推送」按钮误操作 | 全员公告需二次确认 + 限频 |
