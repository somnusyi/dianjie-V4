# 双人协作部署规范

> 2026-05-14 立。两个独立 fork (litianxue18-ops + somnusyi) 反复互相覆盖代码后,
> 改成 single source of truth + git 版本检查 + 服务器锁。

## 协作模型

```
github.com/somnusyi/dianjie-V4         ← 唯一源 (single source of truth)
   ↑ push                                ↑ push
你 (litianxue18-ops)                  同事 (somnusyi 本人)
↓ pull                                  ↓ pull
本地仓库 + worktree                   他的本地仓库
```

**关键约定:**
- 任何人不再用自己的 fork 作为 origin, 一律推到 somnusyi/dianjie-V4
- 部署只在 worktree 目录跑, 不污染主仓库 dev
- 服务器锁机制: 一次只能一个人部署
- 版本检查: 服务器永远拒绝"基于过时代码"的部署

## 一次性 setup (每个人配置一次)

### A. 主仓库 remote

```bash
cd <你的 dianjie-V4 目录>
git remote -v
# 应该看到 origin → somnusyi/dianjie-V4 (如不是, 改过来)
git remote set-url origin https://github.com/somnusyi/dianjie-V4.git
```

### B. 部署 worktree (只你那台机器需要, 同事不需要重复)

```bash
cd <你的 dianjie-V4 主仓库>
git worktree add --detach ../dianjie-V4-deploy main
cd ../dianjie-V4-deploy
pnpm install --frozen-lockfile
pnpm --filter @dianjie/db exec prisma generate
```

worktree 跟主仓库共享 .git 但 node_modules/.next 独立, 主仓库 dev 跑着的时候部署不打扰。

## 日常工作流

### 1. 改代码 (主仓库)

```bash
cd <主仓库>
pnpm --filter @dianjie/api --filter @dianjie/web -r --parallel dev
# api → :4444, web → :3200
# 改代码, HMR 自动生效
```

### 2. 提交 (主仓库)

```bash
git pull --rebase origin main    # 先拉同事最新, rebase 自己改动
git add ...
git commit -m '...'
git push origin main             # 推到 somnusyi 唯一源
```

冲突解决遵循正常 git flow, 改完 add + continue。

### 3. 部署 (worktree, 用脚本)

```bash
cd ~/Desktop/dianjie-V4/dianjie-V4-deploy
export V4_SSH_PASSWORD='xxx'
./scripts/deploy-worktree.sh
```

脚本会自动做:
1. ✅ 抢占服务器锁 (`/app/dianjie-v4/.deploy-lock`)
2. ✅ git fetch origin + worktree 切到最新 commit
3. ✅ 版本检查: 服务器已部署 commit 必须是当前 origin/main 的祖先,否则拒绝
4. ✅ pnpm install + prisma generate + tsc + next build
5. ✅ 产物校验 (cmb.js 存在 / cmbRoutes 注册)
6. ✅ rsync api dist + web standalone + static
7. ✅ pm2 reload api + web
8. ✅ 4 项部署后验证 (ls + grep + md5 + curl 关键路由)
9. ✅ 写入新的 .deployed-commit
10. ✅ 释放锁 (trap EXIT 保证失败也释放)

任意一步失败立刻停止, 不会留下中间状态。

## 为什么这套机制能杜绝代码冲突

| 之前的事故 | 现在如何阻止 |
|---|---|
| 你 push 后,同事基于过期代码部署 → 覆盖你的代码 | 脚本第 3 步: 服务器 `.deployed-commit` 不是 origin/main 祖先 → 拒绝部署 |
| 两人同时部署互相覆盖 | 脚本第 1 步: 服务器锁 (`mkdir .deploy-lock`), 抢不到锁直接退出 |
| 部署完才发现 cmb 路由 404 | 脚本第 7 步: 4 项实测 (ls + grep + md5 + curl), 任意失败 abort |
| 主仓库 dev 部署期间被 build 污染 | worktree 物理隔离 node_modules + .next |

## 死锁恢复

如果脚本异常崩了 (Ctrl+C 杀死前 trap 没执行, 或者机器掉电), 锁文件会留在服务器上。
表现: 下次部署提示 "别人正在部署"。

排查 + 解锁:

```bash
# 1. 看锁里是谁
ssh root@116.62.32.162 'cat /app/dianjie-v4/.deploy-lock/owner'
# 2. 跟那个人确认他不在部署中
# 3. 手动清理
ssh root@116.62.32.162 'rm -rf /app/dianjie-v4/.deploy-lock'
```

## 服务器配置 drift 风险 (单独提醒)

服务器上有几样东西**不在 git 里**, 谁手改了别人就不知道:
- `/app/dianjie-v4/.env` — 数据库密码 / CMB 私钥 / Sentry DSN 等
- `/app/dianjie-v4/ecosystem.config.cjs` — pm2 进程定义 (本来应该跟 git 一致, 但服务器上可能被人手改过)
- `/etc/nginx/sites-enabled/dianjie-v4` — nginx 反代规则
- `/app/dianjie-v4/apps/api/scripts/*.js` — ad-hoc 调试脚本

**约定**: 改这些前在群里说一声 + 把改动写到 docs/CHANGELOG-server.md 里。
